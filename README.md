# Factory Dashboard

**Live Demo:** [https://factorydashboard-d5aw.onrender.com](https://factorydashboard-d5aw.onrender.com)

Factory Dashboard is a production-style, full-stack web application designed to ingest, process, and visualize AI-generated computer vision events from manufacturing factory floors. It tracks worker productivity, workstation utilization, and production counts in real-time.

---

## Architecture Overview (Edge → Backend → Dashboard)



The system follows a streamlined Edge-to-Cloud architecture designed for simplicity and immediate feedback:

1. **Edge (Simulated):** AI-powered CCTV cameras process video streams, extract structured events (`working`, `idle`, `product_count`), and transmit them via HTTP POST requests as JSON payloads.
2. **Backend (FastAPI):** A high-performance Python backend receives the events, validates the schema using Pydantic, and handles database transactions.
3. **Database (SQLite/SQLModel):** Acts as the central repository for metadata and time-series event data. SQLite is used for local portability and zero-config setup, but SQLModel (built on SQLAlchemy) allows a seamless migration to a production database like PostgreSQL.
4. **Dashboard (HTML/CSS/JS):** A lightweight, zero-build frontend. It uses Vue 3 via CDN for reactive data binding and Tailwind CSS via CDN for styling. It is served directly by the FastAPI backend using `StaticFiles`, providing a unified, single-port deployment.

---

## Database Schema

The database consists of three core tables:

| Table | Columns | Description |
| :--- | :--- | :--- |
| **Worker** | `worker_id` (PK), `name` | Stores metadata for the 6 sample factory workers. |
| **Workstation** | `workstation_id` (PK), `name` | Stores metadata for the 6 sample factory workstations. |
| **Event** | `id` (PK), `timestamp`, `worker_id` (Idx), `workstation_id` (Idx), `event_type`, `confidence`, `count` | Persists every AI camera event. |

*Note: The database is automatically seeded on startup with 4 hours of historical data to ensure the dashboard populates with meaningful metrics immediately on the first run.*

## Metric Definitions

To calculate productivity without complex chronometric state tracking, metrics are derived using the following logic:

* **Utilization Percentage:** Calculated as the ratio of `working` events to total time-state events (`working` + `idle`). `product_count` events are excluded from the denominator as they are instantaneous outputs, not continuous states.
  * *Formula:* `(Working Events / (Working Events + Idle Events)) * 100`
* **Total Units Produced:** The sum of the `count` column for all events where `event_type == 'product_count'`.
* **Global/Factory Metrics:** Aggregates the above logic across all workers to provide a factory-wide utilization and production score.

---

## Assumptions and Tradeoffs

* **Event-Based Time Approximation:** Instead of calculating strict `time_end - time_start` durations (which is fragile if a single event drops over the network), utilization assumes that the edge cameras emit state events at a relatively constant frequency. The ratio of these events accurately reflects time spent in each state.
* **Instantaneous Production:** Production events are treated as independent triggers that do not interrupt or override the worker's current `working` or `idle` state.
* **Zero-Build Frontend:** To minimize friction for evaluators, the frontend bypasses Node.js/npm build steps entirely. The simple HTML/CSS/JS structure makes the UI incredibly lightweight and easy to serve directly via Python.

## System Design & Scalability Responses

### 1. Handling Edge Anomalies
* **Intermittent Connectivity:** The edge application (camera side) should implement a local buffer (e.g., a local SQLite database or a lightweight file queue). If the network drops, events are written locally. Upon reconnection, the edge pushes the buffered events to the backend in bulk.
* **Duplicate Events:** To prevent double-counting, the edge should generate a unique UUID for every event based on the camera ID and precise timestamp. The backend database would use this UUID as a primary key with `ON CONFLICT DO NOTHING` (or `INSERT IGNORE`) logic.
* **Out-of-order Timestamps:** The backend strictly relies on the `timestamp` payload generated by the camera, *not* the server ingestion time. Because time-series aggregations query via the payload's original timestamp, out-of-order ingestion does not impact historical metric accuracy.

### 2. MLOps (Versioning, Drift, Retraining)
* **Model Versioning:** The ingestion payload schema should be updated to include a `model_version` string. This allows the backend to tag every event with the specific AI model that generated it.
* **Detecting Model Drift:** We track the average `confidence` score of ingested events over time. If the rolling average confidence drops below a threshold (e.g., < 0.85), or if the ratio of 'working' vs 'idle' drastically shifts from historical baselines, an alert is triggered indicating potential data or concept drift.
* **Triggering Retraining:** A feedback loop would be created where low-confidence frames are securely saved at the edge. A background process uploads these edge-cases to a cloud storage bucket, triggering an automated ML retraining pipeline to fine-tune the model on the new data.

### 3. Scalability (5 to 100+ Cameras to Multi-site)
* **100+ Cameras:** * *Database:* Migrate from SQLite to a time-series optimized database like TimescaleDB or PostgreSQL. 
  * *Ingestion:* Introduce a high-throughput message broker (like Apache Kafka or RabbitMQ) between the Edge and the Backend to queue incoming HTTP requests and prevent dropping events during traffic spikes.
* **Multi-site:**
  * Update the schema to include a `site_id` or `factory_id`.
  * Deploy edge gateways (e.g., AWS IoT Greengrass) at each physical location to aggregate camera data locally before sending it over the WAN to the cloud.
  * Containerize the backend and deploy it on a managed Kubernetes cluster to auto-scale based on CPU/Memory utilization across different regions.
